# 怪物成长与掉落设计

## 背景

当前项目是一个 Phaser + TypeScript + Vite 实现的 2D battle royale 原型。现有架构已经把核心规则放在 `src/game/simulation/state.ts`，Phaser 场景 `src/phaser/scenes/BattleScene.ts` 主要负责渲染、动画、特效和输入适配，HUD 由 `src/ui/hud/HudController.ts` 通过 DOM 渲染。

当前已有能力包括：

- 玩家与机器人对战。
- `bat` 和 `slime` 两类 PVE 怪物。
- 拾取物系统，包括弹药、医疗包、护盾、步枪、霰弹枪和金币。
- 击杀计数、PVE 击杀计数、生命值、护盾和物品栏 HUD。
- 怪物动画资源和资产 manifest。

当前缺口是：击杀怪物只有计数反馈，没有成长、掉落或明确战术价值。玩家缺少主动找怪、绕路打怪或冒险抢资源的理由。

## 目标

本次设计目标是提升局内可玩性，同时保持快节奏吃鸡主线。

具体目标：

- 让击杀怪物有即时收益。
- 增加怪物类型，让地图上的 PVE 压力更丰富。
- 避免玩家只刷怪，不参与 battle royale 对抗。
- 保持规则层和渲染层分离。
- 控制实现范围，第一版不引入完整装备、背包、稀有度、词条系统。

## 非目标

第一版不做以下内容：

- 永久成长或局外养成。
- 装备词条、装备品质、背包管理、装备替换界面。
- Boss 战或多阶段事件。
- 大规模 AI 重写。
- 新地图或地图生成系统。
- 复杂技能树或升级三选一系统。

这些内容可以作为后续版本，但不进入本次实现计划。

## 推荐方案

采用“轻量升级 + 控制掉落”的混合方案。

击杀怪物获得经验值。玩家最多升到 5 级，每级只提供小幅属性收益。怪物死亡时根据类型生成少量补给掉落，例如弹药、金币、护盾或医疗包。精英型怪物掉落更稳定，但生成数量和收益必须受控，避免刷怪收益超过击败其他玩家或抢地图资源。

不推荐第一版直接做完整装备系统。当前项目没有装备栏、词条、稀有度和装备比较界面。直接引入装备会扩大到 HUD、数据结构、资产、平衡和测试多个方向，风险和成本明显高于当前需求。

## 方案取舍

### 方案 A：只做击杀升级

优点：

- 实现简单。
- 反馈直接。
- 容易在 HUD 中表达。

缺点：

- 容易鼓励玩家站桩刷怪。
- 如果升级收益过强，会破坏 battle royale 的紧张节奏。
- 如果升级收益过弱，击杀反馈又不明显。

适用场景：

- 想快速验证“击杀怪物有成长反馈”时可用。

### 方案 B：只做装备掉落

优点：

- 惊喜感强。
- 玩家会更主动争夺怪物资源点。
- 长期可扩展空间大。

缺点：

- 需要装备模型、稀有度、掉落表、装备 UI 和数值平衡。
- 容易让项目从 battle royale 原型变成刷装游戏。
- 第一版改动范围偏大。

适用场景：

- 后续想把游戏转向刷装收集或肉鸽射击时再考虑。

### 方案 C：轻量升级 + 控制掉落

优点：

- 击杀怪物既有成长反馈，也有即时资源反馈。
- 可以复用现有拾取物系统。
- 对 HUD 和 simulation 的改动可控。
- 更符合“快节奏吃鸡”的目标。

缺点：

- 需要额外平衡经验阈值和掉落权重。
- 怪物类型增加后，需要补充资产和动画。

结论：

第一版采用方案 C。

## 成长系统

新增 `progression` 状态，挂在 `GameState` 上。

字段建议：

- `level`: 当前局内等级，范围 1 到 5。
- `xp`: 当前等级内经验。
- `xpToNext`: 下一等级所需经验。
- `totalXp`: 本局累计经验，用于 HUD 或结算。

经验阈值建议：

- Level 1 -> 2: 30 XP
- Level 2 -> 3: 55 XP
- Level 3 -> 4: 85 XP
- Level 4 -> 5: 120 XP

等级收益建议：

- 每级增加 `5` 点 `maxHealth`，最高从 `100` 到 `120`。
- 每级提供 `4%` 玩家伤害加成，最高 `16%`。
- 升级时可以恢复少量生命值，例如 `12` 点，但不能超过新的 `maxHealth`。

设计理由：

- `maxHealth` 提供明确成长感。
- 小幅伤害加成让击杀怪物更有价值，但不会让玩家快速碾压其他角色。
- 升级回血让玩家愿意冒险打怪，但回血量必须小于医疗包收益。

## 掉落系统

新增一个掉落决策函数，放在 simulation 层。

建议职责：

- 输入死亡怪物、击杀来源、当前 game state。
- 根据怪物类型和权重决定是否生成 pickup。
- 用现有 `PickupType` 生成实体，不新增复杂装备实体。
- 生成位置靠近怪物死亡点，并做轻微偏移。

基础规则：

- 只有玩家击杀怪物时，才触发经验奖励和主要掉落。
- 机器人击杀怪物可以不触发掉落，或者只触发低价值掉落。第一版建议不触发，避免地图资源被机器人提前刷掉。
- 风暴击杀怪物不触发奖励。

掉落建议：

| 怪物类型 | XP | 掉落倾向 |
| --- | ---: | --- |
| `bat` | 12 | 低概率 `ammo` 或 `coin` |
| `slime` | 18 | 中概率 `shield` 或 `coin` |
| `boar` | 20 | 中概率 `ammo`，低概率 `medkit` |
| `spitter` | 22 | 高概率 `ammo`，低概率 `shield` |
| `brute` | 38 | 必掉 `coin`，中概率 `shield` 或 `medkit` |

掉落上限：

- 普通怪每次最多生成 1 个拾取物。
- `brute` 最多生成 2 个拾取物。
- 不从普通怪直接掉 `rifle` 或 `shotgun`。
- 如果后续需要武器掉落，只允许精英或事件怪低概率掉弹药包或武器补给。

## 怪物设计

第一版保留现有 `bat` 和 `slime`，新增 3 种怪物。

### `bat`

定位：

- 快速骚扰怪。
- 低生命值，高移动速度。
- 主要作用是打断玩家走位和压迫远程瞄准。

行为：

- 发现近距离玩家后追击。
- 近身造成低伤害。
- 可穿越水域，保持当前地图差异。

### `slime`

定位：

- 慢速近战怪。
- 中生命值。
- 通过周期性脉冲移动制造近身压力。

行为：

- 保持现有跳跃/脉冲式移动。
- 近身造成中低伤害。
- 掉落更偏护盾和金币。

### `boar`

定位：

- 冲锋怪。
- 中生命值，中高移动威胁。
- 强迫玩家横向移动。

行为：

- 进入攻击距离后锁定方向短暂冲锋。
- 冲锋期间速度高，但转向弱。
- 撞到玩家造成中等伤害和轻微击退。

第一版可以用简化实现：

- 不新增复杂状态机，只在 `aiThinkMs` 到期时，如果目标距离合适，提高短时间移动速度。
- 后续再拆成显式 `windup` / `charge` / `recover` 状态。

### `spitter`

定位：

- 远程压制怪。
- 低到中生命值。
- 通过慢速弹体制造区域压力。

行为：

- 保持与玩家中距离。
- 周期性发射低速 PVE 弹体。
- 弹体伤害低于玩家武器，主要用于迫使移动。

第一版可以复用 projectile 实体，但需要标记 `ownerId` 和较低伤害，避免和玩家武器规则混淆。

### `brute`

定位：

- 精英资源怪。
- 高生命值，低速度。
- 高 XP 和更稳定掉落。

行为：

- 缓慢追击最近玩家。
- 近身造成较高伤害。
- 不应大量生成，避免地图变得拥挤。

生成建议：

- 初始只放 1 个 `brute`。
- 普通怪数量可以增加，但 `brute` 应该维持稀缺。

## 怪物数量与地图节奏

现有初始怪物数量是 5 个。第一版建议增加到 10 到 12 个，但按区域分布，避免出生点附近立即被围攻。

建议分布：

- 西侧和西南：`bat`、`boar`
- 中央和南部：`slime`
- 东北和东侧：`spitter`
- 靠近高价值区域：`brute`

节奏规则：

- 前 20 秒内，PVE 可以攻击玩家，但不应该和机器人战斗强绑定。
- 前 20 秒内，怪物对机器人伤害继续受控，避免机器人被怪物过早清掉。
- 怪物不需要无限刷新。第一版只做初始布置，避免引入刷怪节奏和平衡复杂度。

## 数据结构影响

### `GameState`

新增：

- `progression`

建议结构：

```ts
export interface ProgressionState {
  level: number;
  xp: number;
  xpToNext: number;
  totalXp: number;
}
```

### `EntityState`

扩展：

- `pveType` 从 `"bat" | "slime"` 扩展为 `"bat" | "slime" | "boar" | "spitter" | "brute"`。
- 可选增加 `aiState` 或 `attackCooldownMs`，但第一版尽量复用 `aiThinkMs`、`touchCooldownMs` 和 `fireCooldownMs`。

### `GameEvent`

可以扩展事件类型：

- `level-up`
- `xp`

第一版更推荐只新增 `level-up`，避免 HUD 被频繁 XP 事件刷屏。HUD 可以直接读取 `state.progression` 显示经验条。

## 渲染与 HUD

### Phaser 场景

`BattleScene` 继续只做状态适配：

- 根据 `pveType` 选择对应纹理和动画。
- 渲染 `level-up` 特效。
- 继续保持规则逻辑在 simulation 层。

### HUD

HUD 增加：

- 玩家等级，例如 `LV 3`。
- XP 进度条。
- 升级时可以通过事件展示短暂视觉反馈。

HUD 不新增装备栏或装备比较界面。

## 资产策略

新增怪物需要 2D sprite 资源。

遵循 sprite pipeline：

- 每个新怪物先确定一张种子帧。
- 使用整条动画一次性生成，避免逐帧漂移。
- 每条动画统一透明背景、固定帧尺寸、底部中心锚点。
- 资源进入 `public/assets/enemies/<enemy-id>/`。
- manifest 使用稳定 key，例如 `enemy-boar-run`、`enemy-spitter-idle`、`enemy-brute-walk`。

第一版最小动画集：

| 怪物 | 动画 |
| --- | --- |
| `boar` | `run`, `charge`, `hurt` |
| `spitter` | `idle`, `spit`, `hurt` |
| `brute` | `walk`, `slam`, `hurt` |

如果正式资产尚未完成，规则验证阶段可临时复用现有怪物动画；合入正式玩法前必须替换为新怪物正式资源。

## 测试计划

新增或更新 `src/game/simulation/simulation.test.ts`。

推荐测试：

- 玩家击杀 PVE 后获得 XP。
- XP 达到阈值后升级，`level`、`xp`、`xpToNext` 正确更新。
- 升级会提高 `maxHealth`，并且当前生命不超过 `maxHealth`。
- 玩家伤害加成只影响玩家造成的伤害，不影响机器人和风暴伤害。
- 玩家击杀不同怪物时生成符合类型的掉落。
- 机器人或风暴击杀怪物不会给玩家 XP。
- 新增怪物类型在 simulation 中可创建、移动、攻击。

更新 `src/game/assets/asset-manifest.test.ts`：

- 新怪物 spritesheet 文件存在。
- 新怪物 spritesheet 尺寸和帧数正确。
- 新怪物资源保留透明通道。

运行命令：

```bash
npm test
npm run typecheck
npm run build
```

## 风险与缓解

风险：升级收益太强，玩家刷怪后碾压其他角色。  
缓解：等级上限设为 5，单级收益小，普通怪 XP 较低。

风险：怪物数量增加导致地图过于拥挤。  
缓解：初始数量控制在 10 到 12 个，不做无限刷新。

风险：掉落过多导致资源膨胀。  
缓解：普通怪最多掉 1 个拾取物，精英怪最多 2 个。

风险：新怪物 AI 复杂度过高。  
缓解：第一版复用现有 AI 字段，只有必要时再引入显式状态机。

风险：新增资产拖慢实现。  
缓解：规则层先支持新怪物类型，正式资源可按 sprite pipeline 分批补齐。

## 推荐实施顺序

1. 新增 progression 状态和经验阈值计算。
2. 在 PVE 死亡时发放 XP 和升级。
3. 新增掉落决策和 pickup 生成。
4. 扩展 PVE 类型和初始怪物分布。
5. 增加 HUD 等级与 XP 条。
6. 增加新怪物资源 manifest。
7. 补充 simulation 和 asset manifest 测试。
8. 运行测试、类型检查、构建和浏览器验证。

## 验收标准

- 击杀怪物后玩家 XP 增加。
- 达到阈值后玩家等级提升，属性变化可观察。
- 怪物死亡会按类型掉落合理补给。
- 新怪物至少有可区分的行为定位。
- HUD 能展示等级和经验进度。
- 原有拾取物、武器、风暴、胜负判定不回归。
- `npm test`、`npm run typecheck`、`npm run build` 通过。
